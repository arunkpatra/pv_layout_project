# Gate — S8.8: State architecture cleanup

**Status:** Passed 2026-04-24
**Spike:** S8.8 — State architecture cleanup
**Spike definition:** [`docs/SPIKE_PLAN.md`](../SPIKE_PLAN.md) → S8.8
**Driving rationale:** By S9 we'll have four state paradigms in play. Lock the policy now via [ADR-0003](../adr/0003-state-architecture.md), and refactor any existing scatter so S9–S15 builds on a clean foundation.

---

## Goal

Land [ADR-0003 — State architecture](../adr/0003-state-architecture.md) and execute the cleanup it implies: introduce Zustand as the cross-component client-state layer, migrate any existing `useState` that ought to be Zustand, establish the `state/` directory + slice convention, and prove the harness with one slice that S9 will then build on.

## In scope

### Adopt Zustand + state directory

- New deps: `zustand` (with `subscribeWithSelector` middleware enabled by default).
- New directory: `apps/desktop/src/state/`.
- Convention file: `apps/desktop/src/state/README.md` — one-page summary mirroring ADR-0003 §"Conventions". Copy-paste from ADR for proximity.
- Central `apps/desktop/src/state/queryKeys.ts` — TanStack Query key builders.

### Migrate `project` state to Zustand

The current `project` state in `App.tsx` is the cleanest "should-have-been-Zustand-from-the-start" candidate:
- Read by: MapCanvas (via memoised `projectGeoJson`), TopBar (filename for breadcrumb), StatusBar (counts), InspectorSkeleton (projectLoaded toggle).
- Written by: `handleOpenKmz`.
- Will be read by S9's InputPanel + SummaryPanel + GenerateButton.

Action:
- Create `apps/desktop/src/state/project.ts` — Zustand slice with `kmz`, `fileName`, `setProject(kmz, fileName)`, `clearProject()`.
- Replace `useState<{ kmz, fileName } | null>` in `App.tsx` with the Zustand hook.
- MapCanvas and friends read via selectors (`useProjectStore(s => s.kmz)`) so unrelated state churn doesn't re-render them.

### Establish the layout-params + layout-result slices (S9-bound, but introduced here)

- `apps/desktop/src/state/layoutParams.ts` — empty Zustand slice with the params shape from `LayoutParameters` (sidecar schema), default values matching pydantic defaults, action `setParam(key, value)` and `resetToDefaults()`. **Schema defined; no UI consumers yet.** S9 connects react-hook-form to this slice.
- `apps/desktop/src/state/layoutResult.ts` — empty Zustand slice with `result: LayoutResult[] | null`, action `setResult` and `clearResult`. **No consumers yet.** S9 sets it on Generate success.

These are placeholders that S9 fills with UI; living in this spike's scope so S9 doesn't introduce them mid-feature.

## Out of scope

- **`paletteOpen`, `inspectorOpen`, `toolRailOpen`, `units`, dialog open flags** — these stay `useState`. They're ephemeral per-shell UI state; no siblings need them. Migrating would be over-engineering and adds re-render cost.
- **`sidecarPhase`, `bootKeyLoaded`, `savedKey`, `pendingKey`** — these are bootstrap-flow state, lived only during the initial `<App>` mount. Could become a single Zustand "boot" slice but the lifecycle is so contained that `useState` is fine. Re-evaluate if any of these need to be read outside `App.tsx` (none do today).
- **TanStack Query refactoring** — entitlements query is fine. Just adopt the `queryKeys.ts` convention going forward; don't rewrite working code.
- **MapCanvas refs** (`mapRef`, `propsRef`, `lastBoundariesKey`, `lastAppliedStyleUrl`) — already correctly scoped per ADR-0003's "imperative handles" category. No change.
- **Adding any S9 feature** — InputPanel, SummaryPanel, layout mutation. Only the empty stores land here.

## Notes / decisions

- **Zustand v5** — current major. API stable. Subscribe-with-selector middleware ships in core; no extra dep.
- **No `persist` middleware in this spike.** Theme already persists via its own `localStorage` wrapper; no other slice needs persistence yet. Add per-slice when justified.
- **No `immer` middleware.** Slice action payloads are small; immutable spreads are fine. Add if/when a future slice has deep nested updates.
- **The S9 connection point** is already designed: `react-hook-form` initializes its `defaultValues` from `useLayoutParamsStore.getState()` and writes back via `useLayoutParamsStore.setParam` on debounced change. Form lifecycle (touched / errors / submit) lives in RHF; persisted snapshot lives in Zustand. Single source of truth, clear ownership.
