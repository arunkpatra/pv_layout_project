# V2 S3 Buckets — bucket layout, IAM, lifecycle, CORS

**Row:** B5 in [`post-parity-v2-backend-plan.md`](../post-parity-v2-backend-plan.md)
**Tier:** T3 (spike)
**Date:** 2026-04-30
**Status:** Phase A code landed. Phase B (AWS provisioning + Vercel env wiring + round-trip verification) pending.

This memo captures the architectural decisions for V2's blob-storage layer. It is the source of truth for B5; B6 / B7 / B16 / B14 (orphan cleanup deferral) all consume what's decided here.

## 1. Decision summary

| Question | Decision |
|---|---|
| Provider | **AWS S3**, account `378240665051`, region `ap-south-1`. Existing infra. |
| Bucket count | **One per environment** (3 total). KMZ + run results live in the same bucket, separated by key prefix. |
| Bucket names | `solarlayout-local-projects`, `solarlayout-staging-projects`, `solarlayout-prod-projects` |
| IAM | **Extend existing `renewable-energy-app-s3` inline policy** on user `renewable-energy-app`. No new IAM principal. |

| Env var | `MVP_S3_PROJECTS_BUCKET` (added to `apps/mvp_api/src/env.ts`, optional with graceful degradation). |

## 2. Why AWS S3 (not Vercel Blob, not R2)

The V2 plan v1.0 originally said "Vercel Blob recommended". The infra audit (`pv_layout_project/docs/post-parity/discovery/2026-04-29-003-renewable-energy-infra-audit.md`) showed why that was wrong:

- The `renewable-energy-app` IAM user already has S3 read/write/delete/list across 6 prod buckets in `ap-south-1`.
- The existing AWS access key is wired into Vercel envs for `mvp_api` and into local `.env`.
- `apps/mvp_api/src/lib/s3.ts` already contains the SDK + presigner setup.
- Adding Vercel Blob means a new credential, new SDK, new test mocks, split observability, and an extra service in the dependency surface — all for zero operational gain because we already have S3.

V2 reuses S3. R2 (Cloudflare) is not under consideration; it would inherit the same problems as Vercel Blob plus a region-locality concern (S3 in `ap-south-1` is geographically close to the Indian customer base; R2 is global with no regional pinning yet).

---

## 3. Why one bucket per env (not separate KMZ + run-results buckets)

Both KMZ uploads and run-result artifacts:
- Are user-scoped artifacts.
- Have identical IAM grants.
- Need the same lifecycle treatment (orphan cleanup tied to soft-deleted Project / Run).
- Are accessed via presigned URLs only — no bucket-level public ACL distinction.

One bucket means one IAM policy entry, one lifecycle rule set, one set of CORS settings (none, in our case), one set of metrics / alarms. Separating them would split ops surface area for no gain.

Separation lives in the **key prefix**, not the bucket:

```
projects/<userId>/<projectId>/kmz/<sha256>.kmz
projects/<userId>/<projectId>/runs/<runId>/layout.json
projects/<userId>/<projectId>/runs/<runId>/energy.json
projects/<userId>/<projectId>/runs/<runId>/exports/<filename>
```

If we ever need to grant differential access, we can layer that with `Condition: { StringLike: { "s3:prefix": "projects/.../kmz/*" } }` IAM clauses without splitting buckets.

---

## 4. Why `solarlayout-` prefix (not `renewable-energy-`)

Existing pattern in this repo is mixed:
- Legacy artifacts buckets use `renewable-energy-{env}-artifacts` (cloud-port era; dormant — see V2 plan §2).
- MVP downloads buckets use `solarlayout-{env}-downloads` (active customer brand; current).

V2 is post-MVP, customer-facing. Using `solarlayout-{env}-projects` aligns with:
1. The active brand — `solarlayout.in`, `api.solarlayout.in`.
2. The downloads bucket family — same prefix, same env naming, ergonomic for ops scripts.
3. The future direction — V2 plan §7 marks the legacy `renewable-energy-` bucket family for decommission.

Trade-off considered: keeping `renewable-energy-` prefix would reduce bucket-naming variation. Rejected because it would entrench a name we're actively retiring.

## 5. Lifecycle — what's in B5 vs deferred

**In B5 (Phase B):**
- `AbortIncompleteMultipartUpload` after 7 days. Standard hygiene; cleans up failed uploads that left dangling parts.

**Explicitly deferred to a later row:**
- Orphan-blob cleanup for soft-deleted Projects + Runs (V2 plan §7).
- Storage-class transitions (Standard → IA → Glacier).
- Object expiry by age.

Rationale: at v1 launch traffic, accumulation is small enough that aggressive lifecycle introduces more risk of accidental data loss than savings. Revisit when prod telemetry shows storage growth becoming material.

---

## 6. CORS — none

The desktop's Rust shell uses `reqwest` (or similar native HTTP client) for direct PUTs to presigned URLs. Native HTTP does not trigger CORS — there is no browser origin. The bucket has no CORS configuration.

If a future surface (e.g. an admin web UI uploading on behalf of a customer) ever needs to PUT directly from a browser, that's the moment to add a scoped CORS rule for the specific origin. v1: no CORS.

---

## 7. Versioning — off

KMZ keys embed `sha256` — same content always lands at the same key, idempotent overwrites are safe. Run-result keys embed `runId` — never overwritten. Both surfaces are content-addressed by design.

Versioning would 2× storage cost for no data-recovery benefit since the source-of-truth (the database row, the KMZ on the user's machine) survives an accidental delete.

If we ever support user-driven KMZ replacement (changing the sha256 on an existing Project — currently disallowed by B13), we'd revisit.

## 8. Helper signature

```ts
export async function getPresignedUploadUrl(
  key: string,
  contentType: string,
  expiresIn = 900,  // 15 min
): Promise<string | null>
```

- Returns `null` when AWS creds or `MVP_S3_PROJECTS_BUCKET` are missing — graceful degradation, mirrors `getPresignedDownloadUrl`.
- Default `expiresIn` is 900s (15 min) intentionally short. Uploads should land quickly; long-lived signed PUTs widen the replay-misuse window.
- `contentType` becomes `Content-Type` on the signed PUT — the desktop client must echo the same header on its actual PUT or S3 rejects the signature. Surface this in B6/B7's response so the client can't drift.

## 9. Phase B — operational steps

These are the AWS-side changes that complete B5. They are **not** in the Phase A commit because they affect production infra; they execute against the live AWS account.

### B-1. Create the three buckets

```bash
for ENV in local staging prod; do
  aws s3api create-bucket \
    --bucket "solarlayout-${ENV}-projects" \
    --region ap-south-1 \
    --create-bucket-configuration LocationConstraint=ap-south-1
done
```

### B-2. Block public access on all three

```bash
for ENV in local staging prod; do
  aws s3api put-public-access-block \
    --bucket "solarlayout-${ENV}-projects" \
    --public-access-block-configuration \
    BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true,RestrictPublicBuckets=true
done
```

### B-3. Apply baseline lifecycle (abort incomplete multipart uploads after 7d)

```bash
cat > /tmp/v2-projects-lifecycle.json <<'EOF'
{
  "Rules": [{
    "ID": "abort-incomplete-multipart-uploads",
    "Status": "Enabled",
    "Filter": {},
    "AbortIncompleteMultipartUpload": { "DaysAfterInitiation": 7 }
  }]
}
EOF

for ENV in local staging prod; do
  aws s3api put-bucket-lifecycle-configuration \
    --bucket "solarlayout-${ENV}-projects" \
    --lifecycle-configuration file:///tmp/v2-projects-lifecycle.json
done
```

### B-4. Apply the updated inline IAM policy

The Phase A commit updated the policy file at `docs/iam-policy-re-app-s3.json` to add the new `V2ProjectsReadWrite` and `V2ProjectsList` statements. Apply it:

```bash
aws iam put-user-policy \
  --user-name renewable-energy-app \
  --policy-name renewable-energy-app-s3 \
  --policy-document file://docs/iam-policy-re-app-s3.json
```

### B-5. Wire `MVP_S3_PROJECTS_BUCKET` into runtime envs

- **Local** (`renewable_energy/.env`): `MVP_S3_PROJECTS_BUCKET=solarlayout-local-projects`
- **Staging** (Vercel project `mvp_api`, environment `staging`): `MVP_S3_PROJECTS_BUCKET=solarlayout-staging-projects`
- **Production** (Vercel project `mvp_api`, environment `production`): `MVP_S3_PROJECTS_BUCKET=solarlayout-prod-projects`

For the Vercel side, the env var is set via `vercel env add` or the dashboard. After adding production, the next manual `platform-deployment` workflow_dispatch picks it up.

### B-6. Round-trip smoke test

Local script (uses prod creds in `.env.production`):

```ts
// scripts/b5-presigned-roundtrip.ts (don't commit; ephemeral)
import { getPresignedUploadUrl, getPresignedDownloadUrl } from "apps/mvp_api/src/lib/s3.js"

const key = `projects/_smoke/_${Date.now()}/test.txt`
const putUrl = await getPresignedUploadUrl(key, "text/plain", 300)
await fetch(putUrl!, {
  method: "PUT",
  headers: { "Content-Type": "text/plain" },
  body: "hello B5",
})
// Then sign a GET on the same key via getPresignedDownloadUrl variant
// pointing at the projects bucket and assert the body round-trips.
```

`getPresignedDownloadUrl` currently signs against `MVP_S3_DOWNLOADS_BUCKET`; for the round-trip smoke we either (a) generalise it to take a bucket arg, or (b) use `aws s3 cp s3://...` to fetch back. Option (b) is fine for the one-off smoke; B17's run-result download endpoint will need (a) eventually.

## 10. Trade-offs considered and rejected

**Per-user folder-style bucket policies** — would reduce blast radius if a single user's creds leak. Rejected at v1 because we don't have user-scoped creds; everything goes through `renewable-energy-app`. Worth revisiting if we ever issue per-user STS tokens.

**Separate `renewable-energy-app` access key for V2** — would isolate V2 incidents from legacy. Rejected because the legacy buckets are dormant (V2 plan §2); a single key is easier to rotate and observe.

**Bucket-level encryption with a customer-managed KMS key** — would let us audit decryption. Rejected for v1 (no compliance driver); SSE-S3 default is fine. Revisit if a customer requires KMS.

**Move RDS to `ap-south-1` to match S3** — out of scope for B5 (the infra audit logs this as a separate cross-region issue; the presigned-URL pattern doesn't pay the cross-region tax for blobs).
