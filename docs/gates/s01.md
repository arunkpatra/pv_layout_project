## Known limitations (by design)

- **No HTTP / API surface yet.** The sidecar is just a library at this point. FastAPI wrapping lands in S2.
- **No KMZ → layout end-to-end exercise.** Golden-file tests against real KMZs land in S3.
- **Matplotlib is installed but never used interactively.** `pdf_exporter` uses it in `Agg` mode at runtime; we'll verify that explicitly in S12 when the PDF export endpoint is wired.
- **`ezdxf` emits `PyparsingDeprecationWarning`** on import. Not our code; upstream will fix. Harmless; we can silence with a filter later if it becomes noise.

## Decisions made during this spike

1. **Preserved sub-structure (`pvlayout_core/{core,models,utils}/`)** rather than flattening. This keeps the mapping to PVlayout_Advance 1:1 obvious and makes future diffs trivial if we ever need to port a fix back.
2. **Mechanical import rewrite only.** No reorganization, no renaming, no "while we're here" refactors. The promise of "verbatim except imports" is kept strictly.
3. **`matplotlib` kept as a runtime dep.** It is used by `pdf_exporter`. Removing it would require rewriting ~1,000 LOC of PDF generation. Not worth it; matplotlib's `Agg` backend is headless, fast, and has no UI cost.
4. **`ezdxf` is the DXF writer.** Matches `PVlayout_Advance` exactly.

No ADR triggered — all decisions are execution detail, not architectural.
