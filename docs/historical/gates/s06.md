## Chrome decision — explicit deviation from spec

**Spec (SPIKE_PLAN.md S6):** "`decorations: false` on macOS; re-implement traffic-light positioning".

**What shipped:** `decorations: true` + `titleBarStyle: "Overlay"`. Tauri 2's native Overlay mode keeps macOS traffic lights in their normal position on a borderless titlebar, achieving the same visual result as the spec without writing platform-specific Rust for `NSWindowStyleMaskFullSizeContentView`. Claude Desktop uses the same technique.

**Trade-off:** saves ~50 LOC of Rust + zero risk of traffic-light-repositioning bugs across macOS versions. Cost: we don't have the option to, say, put an inline tab bar in the titlebar area. Not a constraint we need today.

**If you disagree:** say so and I'll flip to `decorations: false` in a follow-up; the cost is real Rust work in `src-tauri/src/window.rs` and ongoing macOS-version risk.

## Gate-blocker resolution log

Two latent defects surfaced during the gate run and were fixed in-spike before close.

### Defect 1 — Window drag inert on macOS

**Symptom:** clicking and dragging the top bar didn't move the window; traffic lights still worked.
**Root cause:** `core:window:default` does not include `allow-start-dragging`. Tauri 2 silently rejected both the native `data-tauri-drag-region` handler and any programmatic `window.startDragging()` call.
**Fix:** added `"core:window:allow-start-dragging"` to [`capabilities/default.json`](../../apps/desktop/src-tauri/capabilities/default.json).
**Cleanup:** removed two tech-debt workarounds accumulated during the bug hunt — a stale `-webkit-app-region` / `app-region` CSS block in `globals.css` (Electron-era pattern not read by Tauri 2) and a JS `mousedown` fallback in `App.tsx` that duplicated Tauri's native injected handler. Ships with a single, canonical mechanism: `data-tauri-drag-region` HTML attribute on the TopBar spans + the ACL permission. Detailed bug hunt preserved at [`S6_DRAG_BUG_HANDOFF.md`](./S6_DRAG_BUG_HANDOFF.md).
**Reference:** [tauri-apps/tauri#9503](https://github.com/tauri-apps/tauri/issues/9503).

### Defect 2 — Status-bar m/ft toggle had no visible flip

**Symptom:** clicking `ft` did not visibly change the segmented toggle; active and inactive states looked identical.
**Root cause:** the base reset in `globals.css` (e.g. `button { color: inherit; background: transparent }`) was unlayered. CSS cascade rule: unlayered declarations beat any declaration inside `@layer`, regardless of specificity. Tailwind v4 emits utilities into `@layer utilities`, so every `bg-[var(…)]` / `text-[var(…)]` on every `<button>` in the app was silently overridden. The units toggle was the most visible symptom because both states inherited the same footer color, making the "flip" invisible.
**Fix:** wrapped the base reset in `@layer base { … }` in [`packages/ui/src/globals.css`](../../packages/ui/src/globals.css). Layer order `base < utilities` is Tailwind's intended discipline.
**Verified:** Playwright against the live dev server confirmed every button's utility-driven bg/color now applies (AP avatar, ToolRail hover, command palette trigger hover, command palette rows, status-bar toggle).

### Incidental — Radix accessibility warning

Added `sr-only` `<DialogPrimitive.Description>` to [`CommandPalette.tsx`](../../packages/ui/src/components/CommandPalette.tsx) to clear the "Missing Description or aria-describedby" warnings Radix emitted in dev.
