## 1. What shipped

Seven additive changes inside a scoped CLAUDE.md §2 exception (ADR 0007, amended mid-spike 2026-04-24 to add Pattern V):

1. **Search-space pruning** inside `_route_ac_cable` (`string_inverter_manager.py`) — caps per WHAT-CHANGED.md's table: A2 / A3 ≤ 8 nearest cols, A4 5 × 5, B main 8 × 8 nearest gaps, E single ≤ 15, E two-waypoint only when |W| ≤ 10. Pattern order and candidate-path geometry preserved.
2. **Dormant instrumentation** in `_route_ac_cable` + `place_string_inverters` gated on `PVLAYOUT_PATTERN_STATS=1`. Per-pattern hit counts + per-cable `_path_ok` call counts emitted to stderr for DC and AC cable loops.
3. **Pattern F quality tagging** — `CableRun.route_quality` ∈ `{"ok", "best_effort", "boundary_violation"}`. Every cable that resolves via Pattern F gets either `"best_effort"` (all segments inside the polygon) or `"boundary_violation"` (at least one segment leaves the polygon). Clean pattern hits keep `"ok"`.
4. **Additive `LayoutParameters` fields** — `ac_termination_allowance_m: float = 4.0`, `dc_per_string_allowance_m: float = 10.0`. Defaults preserve existing numeric behaviour; customer-site tuning no longer needs a code change.
5. **Additive `LayoutResult` fields** — `ac_cable_m_per_inverter: Dict[int, float]`, `ac_cable_m_per_icr: Dict[int, float]`. Grouped AC subtotals for EPC BOMs.
6. **Sidecar wire mirrors** in `schemas.py` + `adapters.py` for the three new model fields in (3)–(5). Round-trip preserved on `/refresh-inverters`.
7. **Pattern V — visibility-graph shortest-path fallback** (ADR 0007 amendment). Inserted between Pattern E and Pattern F. Uses a cached visibility graph on polygon-boundary vertices + Dijkstra (heap-based, Python stdlib `heapq`). The graph is built against a purpose-constructed **route polygon** = `boundary_wgs84` projected to UTM, minus ICR footprints — the physical plant-fence polygon, which includes the perimeter-road area where cables actually lie. `usable_polygon` is still used by patterns A–F (stays close to row gaps); only Pattern V uses `route_poly`. This matters on plants where `usable_polygon` splits into a disjoint `MultiPolygon` under perimeter setbacks — exactly what happens on `phaseboundary2`, where the 15 cables previously routing 34–64 m outside the plant boundary now route inside.

Headless benchmark script: [`python/pvlayout_engine/scripts/debug/time_cable_calc.py`](../../python/pvlayout_engine/scripts/debug/time_cable_calc.py).

### 2.3 Pattern V — how it fixed the 15 boundary-violation cables

**Mid-spike scope extension.** The instrumented pre-port run revealed that spec §3.2's remediation (iii) (loosen A4 cap from 5 × 5 to 8 × 8) would have had zero effect on `phaseboundary2` — the 15 problem cables never enter A4. The remediation was designed around the review-package's peer plant, where A4 was succeeding. Our plant's failure mode is different.

User approved scope extension to option (a) from the industry-routing conversation: add a **visibility-graph shortest-path pattern (Pattern V)** between E and F. ADR 0007 amended 2026-04-24.

**Why Pattern V alone wasn't enough — also had to fix the polygon.** First integration attempt had Pattern V validating against `usable_polygon`. Test showed Pattern V still didn't fire: `usable_polygon` turned out to be a disjoint `MultiPolygon` with 3 sub-pieces (59 k m² + 10 m² + 139 k m²) because perimeter setbacks had split the plant at narrow-neck points. The 15 problem cables route from sub-piece 2 to sub-piece 0 — no inside-`usable` path can exist between disjoint components.

**Fix:** Pattern V validates against a purpose-built `route_poly` instead — the plant boundary (`boundary_wgs84` projected to UTM) minus ICR footprints. This is the **physical plant-fence polygon, contiguous by construction** (unless an ICR sits in a narrow neck, which real EPCs don't do). Obstacle polygons are deliberately NOT subtracted from `route_poly`: they mark "no tables here", not "no cables here" — cables route around / through obstacles at the trench level in real installations. On `phaseboundary2` this gave a single 220,625 m² contiguous route polygon, which Pattern V can navigate freely.

Patterns A–F continue to validate against `usable_polygon` (keeps Manhattan routes close to row gaps — the visual convention). Only Pattern V uses `route_poly`.

**Result on `phaseboundary2`:**

| Before Pattern V | After Pattern V |
|---|---|
| 15 AC cables route 34–64 m outside plant boundary | 0 AC cables outside |
| 15 cables tagged `boundary_violation` | 0 cables tagged `boundary_violation` |
| Pattern F count: 15 | Pattern F count: 0 |
| Pattern V count: (didn't exist) | 15 (all 15 former-F cables) |
| `total_ac_cable_m`: 14,474.8 m (over-counts the outside-detour portions) | 12,361.0 m (physically installable) |

**Generality:** Pattern V is correct for arbitrary polygon shape. The algorithm is textbook: Preparata & Shamos 1985; de Berg et al. *Computational Geometry* ch. 15. Future plants with concavities, narrow necks, or interior obstacles will also route correctly without any per-plant tuning.

## 6. Known limitations

- Cable gauge and voltage drop are **not** computed. S11.5 outputs cable run lengths only. Gauge selection + voltage drop computation are S12 / S13 scope.
- DC cables are modelled **per-table aggregated** (one route × strings-per-table multiplier). Per-string routes are not produced.
- BOM export format is not implemented. Export spikes own export formats.
- The `+4.0 m` / `+10.0 m` allowances are now parameterised but the frontend does not yet expose them in the input panel. Sidecar accepts them in `LayoutParameters`; frontend change is deferred.
- The `_route_ac_cable` function name is arguably misleading since it also handles DC routing. Rename is deliberately deferred to a cleanup spike.

## 7. Decisions locked in S11.5

- **CLAUDE.md §2 is an ADR-gated exception regime**, not an absolute prohibition. This is the first use.
- **Capacity-based inverter-count formula retained** as industry-standard. Checkpoint's tables-based formula rejected.
- **Pattern A → A2 → A3 → A4 → B → C → D → E → F order** preserved as the vendored code had it; no reordering under this exception.
- **Performance target of ≤ 30 s** on the `phaseboundary2` reference plant is the acceptance bar; future plants that exceed this are diagnosed via the same headless script + `PVLAYOUT_PATTERN_STATS` machinery.
- **Pattern F best-effort routes are acceptable** when route segments stay inside the polygon. Boundary-crossing routes are surfaced as `route_quality == "boundary_violation"` and visually distinguished; they do not fail the gate unless the rate exceeds 5 %.
