## What was built

A Tauri 2 desktop app that spawns the pvlayout-engine sidecar, parses its `READY` line, and exposes its host/port/token to a minimal React page. The page fetches `/health` and renders status. Dev and release process-trees both tear down cleanly when the app quits.

### 1. Rust toolchain

Installed **Rust stable 1.95** via rustup (minimal profile, `--no-modify-path`). Lives in `~/.cargo/` and `~/.rustup/`; nothing outside those. Remove anytime with `rustup self uninstall`. Nothing of yours was changed.

Build scripts source `$HOME/.cargo/env` inline where needed; `bundle-sidecar.mjs` looks for `rustc` on PATH and falls back to `~/.cargo/bin/rustc` so contributors don't need to modify their shell profiles unless they want to.

### 2. `apps/desktop/` ‚Äî the Tauri app

**React frontend** (`apps/desktop/src/`):
- `main.tsx`, `App.tsx` ‚Äî minimal, functional. Invokes `get_sidecar_config`, fetches `/health` via `@solarlayout/sidecar-client`, renders one of three states: **booting** (grey), **healthy** (green), **error** (red).
- Placeholder visual only ‚Äî real design system lands in S5.5/S6.
- Splash role is played by the "booting" state during the ~5‚Äì10 s cold-start window; no second window yet (S6 adds one).

**Tauri backend** (`apps/desktop/src-tauri/`):
- `Cargo.toml` ‚Äî Tauri 2.x, `tauri-plugin-shell`, `tokio`, `serde`, `env_logger`, `rand`.
- `tauri.conf.json` ‚Äî single window, 1200√ó800 min 900√ó600, bundle targets `dmg/app/msi/nsis/deb/appimage`, `externalBin: ["binaries/pvlayout-engine"]`.
- `capabilities/default.json` ‚Äî permissions for shell spawn/execute/kill on the main window.
- `src/main.rs` ‚Äî trivial entry; calls `solarlayout_desktop_lib::run()`.
- `src/lib.rs` ‚Äî app builder, state management, `get_sidecar_config` command, `on_window_event` and `RunEvent::Exit` shutdown hooks.
- `src/sidecar.rs` ‚Äî the spawn + READY-parse + shutdown machinery.

**Icons** ‚Äî placeholder set generated from a solid-colour 512√ó512 via `tauri icon`. Real branding lands in S6.

### 3. Sidecar lifecycle (`src/sidecar.rs`)

Branching by `cfg!(debug_assertions)`:

- **Dev** (`bun run tauri dev`): spawns `$REPO/python/pvlayout_engine/.venv/bin/python -m pvlayout_engine.main` with `current_dir` set to the engine directory. `CARGO_MANIFEST_DIR` resolves the path at compile time so we're not hostage to runtime CWD. The `.venv` python is a direct child ‚Äî **no `uv run` wrapper** ‚Äî so signal-kill propagates without orphaning.
- **Release** (`bun run tauri:build`): uses Tauri's own `shell.sidecar("pvlayout-engine")` to resolve the bundled binary inside the `.app` / `.msi` / `.AppImage`.

Both pass two env vars to the sidecar: `PVLAYOUT_SIDECAR_TOKEN` (random 32-char URL-safe) and `PVLAYOUT_PARENT_PID` (Rust's own PID ‚Äî see ¬ß5).

After `Command::spawn()` returns `(rx, child)`, a tokio task reads `CommandEvent::Stdout` lines, looking for `READY {json}` and publishing the resolved `SidecarConfig` to shared state. `CommandEvent::Terminated` or a 45s timeout flip the state to `Failed(msg)`.

The child handle lives in `SidecarState.child: Arc<Mutex<Option<CommandChild>>>`. On `WindowEvent::Destroyed` and `RunEvent::Exit` we `block_on(shutdown())` which `child.kill()`s it.

### 5. Parent-death watchdog ‚Äî the real fix for clean shutdown

**What we hit.** Two layered problems, both discovered by actually running the app:

1. **Dev mode via `uv run`:** `uv run python -m X` keeps `uv` as the parent and python as a grandchild. When Rust's `child.kill()` SIGKILLs `uv`, the python grandchild is orphaned to launchd. Fix: spawn `.venv/bin/python` directly so python is Rust's direct child.
2. **Release mode via PyInstaller onefile:** the onefile bootloader is its own process that forks the actual python child. Python's `os.getppid()` returns the bootloader PID, not Tauri's ‚Äî so a watchdog that just checks "did my parent change" doesn't help when Tauri itself dies (the bootloader is still alive, still the immediate parent).

**The fix.** Rust passes its own PID in `PVLAYOUT_PARENT_PID`. The Python sidecar starts a daemon thread that polls `os.kill(parent_pid, 0)` every 1s; when `ProcessLookupError` fires (or `PermissionError`, which indicates PID reuse), it calls `os._exit(0)`.

This works regardless of process-tree shape:
- Dev: python is direct child of Tauri; watchdog still works (redundant belt-and-braces with SIGTERM).
- Release: python is great-grandchild of Tauri (through the PyInstaller bootloader); watchdog is the *only* thing that catches a SIGKILLed Tauri.

No psutil needed ‚Äî `os.kill(pid, 0)` is the cross-Unix idiom. Windows isn't guarded yet; Tauri's explicit `child.kill()` on window-close covers the normal path, and a psutil probe can be added in a follow-up spike if a Windows-specific orphan ever shows up.

### 6. Build automation

- `scripts/bundle-sidecar.mjs` ‚Äî Node script that copies `python/pvlayout_engine/dist/pvlayout-engine` ‚Üí `apps/desktop/src-tauri/binaries/pvlayout-engine-<target-triple>`. Resolves the triple from `rustc -vV`, with a `~/.cargo/bin/rustc` fallback so shell-profile PATH isn't required. Uses `execFileSync` (no shell) to sidestep any injection surface.
- `bun run bundle:sidecar` invokes it; `tauri.conf.json`'s `beforeBuildCommand` chains it in front of `vite:build`. `bun run tauri:build` is the top-level "give me the installer" command.
- `apps/desktop/src-tauri/binaries/.gitignore` excludes the copied binaries from git (they're build artifacts).

- `sidecar-client` has a real `tsconfig.json` and its `build` is source-only (consumers transpile).

## Known limitations (by design or flagged)

- **Cold-start ~5‚Äì10 s in release**, due to PyInstaller onefile `_MEIPASS` unpack. Currently covered by the "booting" text in the React UI ‚Äî a real splash window lands in S6. This is the known S4‚ÜíS5 concern we agreed to paper over with a splash.
- **No real UI.** S5's React page is one line of status text. S5.5 authors the design system + mocks; S6 implements.
- **No code signing.** The `.app` will trigger Gatekeeper ("unidentified developer") on a clean machine. That's the S14 deliverable; acceptable for local dev.
- **Windows parent-death watchdog is a no-op.** The PVLAYOUT_PARENT_PID check uses `os.kill(pid, 0)` which works on Unix. On Windows, `os.kill(pid, 0)` raises `OSError: [WinError 6]` for invalid handles but doesn't reliably report liveness; will add a psutil-based probe if Windows orphans surface in practice.
- **Icon is a placeholder** (solid off-white + orange circle). Real brand icon lands in S6.

## Decisions made during this spike

1. **Rust installed via rustup with `--no-modify-path`.** Minimal profile, no shell-profile changes, entirely confined to `~/.cargo` and `~/.rustup`. Uninstall anytime with `rustup self uninstall`.

2. **Spawn `.venv/bin/python` directly in dev, NOT `uv run`.** Avoids the orphan-grandchild problem when the Rust parent is SIGKILLed. Trade-off: the `.venv` must exist (so `uv sync` is a prerequisite for `bun run tauri dev`). The watchdog would catch it anyway, but direct spawn is cleaner for the normal path.

3. **`PVLAYOUT_PARENT_PID` env var over `os.getppid()`.** Robust against PyInstaller's bootloader interposition and against any future process-tree reshuffling. One env var > subtle bugs.

4. **Splash is React-rendered text (not a separate window) in S5.** Second-window splash needs a visual design, which S5.5/S6 owns. The functional piece (showing something while sidecar boots) is delivered.

5. **Desktop workspace `build` is frontend-only.** Putting `tauri build` behind a separate script keeps `bun run build` (the repo-root gate) fast. `bun run tauri:build` is the explicit "give me an installer" command.

6. **CORS left permissive on the sidecar.** The real authentication boundary is the bearer token; CORS is a dev-tool affordance. Fine for localhost-only.

No ADR triggered.

## Post-gate fix (follow-up commit)

During the human gate, the release `.app` showed **"Sidecar error: Load failed"** (later, after a scope widening, **"error sending request for url (http://127.0.0.1:<port>/health)"**). Three nested issues had to be resolved:

1. **WebKit cross-origin/ATS blocks plain `fetch()` in release builds.** The WebView serves the React page from `tauri://localhost` and blocks `fetch("http://127.0.0.1:...")`. Dev mode didn't hit this because the page was on `http://127.0.0.1:1420` (Vite), same scheme. Release is `tauri://`, different. **Fix:** added `tauri-plugin-http` (Rust side) and `@tauri-apps/plugin-http` (frontend side); routed the sidecar client's fetch through Rust via `fetchImpl: tauriFetch`.

2. **`http:default` scope pattern had to be `http://127.0.0.1:*/*` (trailing `/*`).** Without the explicit path wildcard, plugin-http rejected URLs before sending ‚Äî that was the "Load failed" before the scope fix. **Fix:** `apps/desktop/src-tauri/capabilities/default.json` now allows `http://127.0.0.1:*/*` and `http://localhost:*/*`.

3. **`READY` was emitted before uvicorn bound the TCP socket.** With the scope fixed, fetch started reaching Rust, but reqwest got connection-refused because the sidecar's `_announce_ready` ran synchronously before `uvicorn.run()` could bind. (FastAPI lifespan startup would still be too early ‚Äî uvicorn runs lifespan startup before socket bind.) **Fix:** `_announce_when_listening()` thread in `pvlayout_engine/main.py` polls `socket.create_connection((host, port))` every 50 ms and only emits `READY` once the connect succeeds. 10-second fallback timeout in case of pathological delays.

These three together produced the timing + transport story that the release `.app` needed. After the fix, the observed timeline is:

```
uvicorn: Started server process
uvicorn: Application startup complete
uvicorn: Uvicorn running on http://127.0.0.1:64978
sidecar READY on 127.0.0.1:64978          ‚Üê only now
GET /health -> 200                         ‚Üê React fetch succeeds
```

**Also bumped in this fix commit:**
- `env_logger` now honors `RUST_LOG` (was hard-filtered to INFO) ‚Äî needed during diagnosis.
- Sidecar stderr pass-through elevated from `debug!` to `info!` so uvicorn's request logs are visible during gate verification.
- React error handler widened ‚Äî "Load failed" alone hid the real cause.

Unchanged from S5 proper: the rest of the gate (window opens native, sidecar lifecycle, dev/release branching, parent-death watchdog, clean shutdown within 1s) remained correct throughout.
