## 1. Context

C3 established `python/lambdas/` with the production Dockerfile + ECR push pipeline + matrix CI workflow. Smoketest Lambda exists as a throwaway demonstrator that C4 deletes when parse-kmz arrives.

C3.5 ships the **cross-runtime local-dev transport pattern** — the contract that lets every cloud Lambda in the arc (parse-kmz, compute-layout, detect-water, compute-energy) be developed and tested on a laptop without a cloud-deploy round-trip. The pattern is load-bearing: every downstream Lambda row (C4/C6/C16/C18) inherits its shape verbatim. Choices made here cascade across the entire arc.

**Why this row exists:** D9 mandates that Lambdas write directly to RDS via psycopg2. Cloud Lambdas can write to staging/prod RDS over the VPC. **AWS-hosted Lambdas cannot reach a developer's local Postgres.** Without a parallel local-HTTP transport, every Lambda code change requires a cloud-deploy round-trip just to be exercised — unworkable for iteration. C3.5 closes that gap.

## 2. Pattern source: journium-bip-pipeline (production-validated)

Pattern adapted directly from `journium-bip-pipeline`, which has been running in production for months as a SQS-shaped Python Lambda with a parallel local HTTP entry. Specifically:

- **Single production Dockerfile** (`apps/journium-bip-pipeline/Dockerfile`) using `public.ecr.aws/lambda/python:3.11` base. NO `Dockerfile.local`.
- **`server.py` lives in source** (`apps/journium-bip-pipeline/src/server.py`).
- **Local dev runs `server.py` natively on the host** via:
  ```json
  "serve": {
    "command": "PYTHONPATH=src uv run python src/server.py",
    "cwd": "apps/journium-bip-pipeline"
  }
  ```
- **NOT in `docker-compose.yml`** — only Postgres lives there. The Lambda runs as a host process; it reaches the compose-Postgres via the host-mapped port.
- **Business logic factored transport-agnostic** (`orchestrator.py:run_pipeline(plain_args)`) so both `lambda_handler.handler(SQS event)` and `server.py` (HTTP body) call the same function after unwrapping their respective transport envelope.

`journium-litellm-proxy` uses a different shape (Dockerfile + Dockerfile.local + docker-compose service) — **explicitly rejected as the bad example.** The `Dockerfile.local` pattern requires duplicate dependency wiring and a docker rebuild for every code change; native `uv run` restarts in ~1s.

**AWS Lambda RIE was explored** as a "single Dockerfile, no server.py" alternative. Rejected per `feedback_proven_over_novel.md`: prefer the proven journium pattern over a technically-nicer alternative that would require new validation effort across hot-reload, fire-and-forget semantics, and `host.docker.internal` portability.

## 3. Six design choices made

### Q1 — server.py route shape

**Per-Lambda shape mirrors that Lambda's cloud trigger type.** Not a runtime config flag. Each Lambda's `server.py` is hand-coded to behave like its cloud trigger.

| Cloud trigger | Local server.py response |
|---|---|
| Sync invoke (parse-kmz, smoketest) | `POST /invoke` returns 200 + handler result (200 on success / 500 on exception) |
| SQS-triggered (compute-layout, detect-water, future compute-energy) | `POST /invoke` returns 202 immediately + spawns `Thread(daemon=True).start(target=process_message)` |

Plus `GET /health` returning `{"ok": true}` on every Lambda (uniform, used by the dev for liveness curl).

**Business logic factored transport-agnostic:** every Lambda's `<purpose>_lambda` package exposes a function (e.g., `process_message(payload: dict) -> dict | None`) that takes plain Python data, not the SQS-event envelope. Both entry points are thin transport unwrappers:

- `lambda_handler.handler(event, context)` — for SQS-triggered: iterates `event["Records"]`, `json.loads(record["body"])`, calls `process_message(body_dict)`. For sync-invoked: `process_message(event)` (the SDK Invoke `Payload` arrives as `event` directly).
- `server.py` — reads HTTP POST body as JSON dict, calls `process_message(body_dict)` (in a thread for async-mode; inline for sync-mode).

The SQS-event shape **never leaks** into local code; the HTTP shape **never leaks** into Lambda code. (litellm-proxy + bip-pipeline pattern.)

**For C3.5 specifically:** smoketest is sync-mode (simplest demonstrator). Its handler IS the business logic — no `process_message` factor needed. C6 (compute-layout) is the first row that exercises the async + factored shape; the README documents the pattern so C6 inherits it cleanly.

### Q2 — `lambda-invoker.ts` API shape

```ts
// apps/mvp_api/src/lib/lambda-invoker.ts
async invoke(purpose: LambdaPurpose, payload: object): Promise<object>     // sync — block for result
async enqueue(purpose: LambdaPurpose, payload: object): Promise<void>      // async — fire-and-forget
```

**Naming rationale:** `invoke` / `enqueue` describe the actual operation in cloud (SDK Invoke vs SQS publish) AND the caller's mental model. Avoids the false JS-sync-vs-async cue that `invokeSync` / `publishAsync` would carry (both methods return Promises). Locality is internal to the util — call sites never branch on env.

**Branching internals:**
- Both methods read `process.env.USE_LOCAL_ENVIRONMENT === "true"`.
- **Local branch:** `fetch(LOCAL_<PURPOSE>_LAMBDA_URL || \`http://localhost:${DEFAULT_PORT[purpose]}\`)`. For `invoke()`: `await response.json()`. For `enqueue()`: don't await body (request fired; response discarded; daemon thread continues handler).
- **Cloud branch (this row):** throw `NotImplementedError("AWS SDK Lambda invoke wired in C4")` (for `invoke`) and `NotImplementedError("AWS SDK SQS publish wired in C7")` (for `enqueue`). Forward-typed stubs.

**Forward-looking — C4 fills `invoke()` cloud branch:** `LambdaClient.Invoke({FunctionName: \`solarlayout-${purpose}-${ENV_SUFFIX}\`, InvocationType: "RequestResponse", Payload: JSON.stringify(payload)})` where `ENV_SUFFIX ∈ {staging, prod}` is read from `process.env.VERCEL_ENV` or equivalent. The function-name format is locked at C3's `solarlayout-<purpose>-<env>` convention (see `python/lambdas/README.md`).

**Forward-looking — C7 fills `enqueue()` cloud branch:** `SQSClient.SendMessage({QueueUrl: <queue-url-for-purpose>, MessageBody: JSON.stringify(payload)})`. Queue URL resolution from env (`SQS_<PURPOSE>_QUEUE_URL`).

### Q3 — Async-mode background execution

`threading.Thread(target=fn, daemon=True).start()`. Matches journium-bip-pipeline's `server.py` verbatim.

- Daemon thread: dies when the parent server.py process is killed (Ctrl-C). No orphan.
- Per-request thread: simplest possible model; no pool to size; matches "one Lambda invocation = one thread of execution" semantics.
- Crash behavior: each handler wraps its own work in `try/except` (per the bip-pipeline orchestrator pattern); uncaught exceptions get logged via `logger.exception(...)` before the thread dies. **DLQ is not emulated locally** — devs see logs; that's the contract.

**For C3.5 specifically:** smoketest is sync; doesn't exercise this. C6 is the first to use it; the README documents the pattern with a code reference to bip-pipeline.

### Q4 — Test approach

**No automated `bun test` for the lambda-invoker contract.** (v1.5 amendment removed this requirement.)

Reasoning: the lambda-invoker is a 30-line transport shim whose cloud branches are `NotImplementedError` stubs that get replaced wholesale at C4 and C7. An automated mock-fetch test verifies mechanical wiring (URL construction + env-switch read + throw) that:
- Gets exercised end-to-end by the manual smoke (curl + mvp_api dev server) anyway,
- Gets rewritten at C4 + C7 — would churn twice within weeks.

Manual smoke is the gate. Evidence captured in PR per §11.2 + `SMOKE-LOG.md` per §11.4.

### Q5 — Dockerfile shape

**Single production Dockerfile per Lambda. NO Dockerfile.local.** (v1.6 amendment.)

Smoketest's existing production Dockerfile (committed in C3) is unchanged. Local dev runs `server.py` natively via `cd python/lambdas/smoketest && uv run python -m smoketest_lambda.server`.

For C6+ Lambdas, the same pattern: production Dockerfile bakes `pvlayout_core` + handler code; local dev runs `server.py` natively from the Lambda's directory.

### Q6 — DB-connection lifecycle for C6+ Lambdas

Per-invocation psycopg2 connection (matches per-invocation Lambda model in cloud). No connection pool in `server.py`.

- `MVP_DATABASE_URL=postgres://mvp:mvp@localhost:5433/mvp_db` in the dev's shell env.
- `server.py`'s handler thread opens a connection, does its work, closes the connection.
- The compose-Postgres at host port 5433 is reachable from the natively-launched Python process — no docker network required.
- For DB-using Lambdas, `pyproject.toml` adds `psycopg2-binary` (or `psycopg[binary]`) as a dep.

**For C3.5 specifically:** smoketest doesn't touch the DB. The pattern is documented in `python/lambdas/README.md` for C6 to inherit.

## 4. Architecture (consolidated)

```
                                  HOST MACHINE
   ┌──────────────────────────────────────────────────────────────────┐
   │                                                                  │
   │   ┌─────────────────┐         ┌─────────────────────────┐        │
   │   │  apps/mvp_api   │         │  python/lambdas/        │        │
   │   │  bun run dev    │         │  smoketest/             │        │
   │   │                 │         │  uv run python -m       │        │
   │   │  USE_LOCAL_     │  HTTP   │  smoketest_lambda.server│        │
   │   │  ENVIRONMENT=   ├────────►│                         │        │
   │   │  true           │  fetch  │  port 4100              │        │
   │   │                 │         │                         │        │
   │   │  lambda-        │         │  smoketest_lambda/      │        │
   │   │  invoker.ts     │         │  ├── handler.py         │        │
   │   └─────────────────┘         │  └── server.py          │        │
   │                               └─────────────────────────┘        │
   │                                                                  │
   │   ┌─────────────────┐                                            │
   │   │ docker-compose: │                                            │
   │   │  mvp_postgres   │  port 5433 → host (for C6+ Lambdas         │
   │   │                 │              that need DB; smoketest       │
   │   │                 │              doesn't)                      │
   │   └─────────────────┘                                            │
   │                                                                  │
   └──────────────────────────────────────────────────────────────────┘
```

**Cloud (unchanged from D24) — TWO distinct paths, one per Lambda mode:**

*Sync path (parse-kmz; `invoke()` call from mvp_api):*
```
                              AWS
   mvp_api on Vercel ─► AWS SDK Lambda Invoke ─► Lambda function
                        InvocationType:           solarlayout-parse-kmz-<env>
                        "RequestResponse"          (single Dockerfile, baked
                                                    via C3 pipeline)
```

*Async path (compute-layout, detect-water, future compute-energy; `enqueue()` call from mvp_api):*
```
                                  AWS
   mvp_api on Vercel ─► AWS SDK SQS ─► SQS queue ──► event source ──► Lambda function
                        SendMessage     solarlayout-   mapping          solarlayout-
                                        <purpose>-     (auto-trigger)   <purpose>-<env>
                                        jobs + DLQ
```

`USE_LOCAL_ENVIRONMENT` is unset / false for both cloud paths. The mode (sync vs async) is intrinsic to the Lambda — locked at deploy time, not at call time. mvp_api picks the verb (`invoke` vs `enqueue`); lambda-invoker.ts routes to the corresponding cloud path.

## 7. Forward-looking notes (for C4+ child sessions)

- **Cloud function-name resolution:** `solarlayout-<purpose>-<env>` per the C3 convention (README at `python/lambdas/README.md`). `<env> ∈ {staging, prod}` (abbreviation; not `-production`). `lambda-invoker.ts`'s cloud branch reads `ENV_SUFFIX` from `process.env.VERCEL_ENV` or equivalent at C4.
- **AWS SDK installs:** `@aws-sdk/client-lambda` added to mvp_api at C4; `@aws-sdk/client-sqs` added at C7. Confirmed NOT yet present (V5 verification).
- **Async pattern first exercise:** C6 (compute-layout) is the first Lambda where `server.py` returns 202 + spawns the daemon thread. README has a forward-pointing note; C6's child session implements the pattern using `journium-bip-pipeline/src/server.py` + `orchestrator.py` as the reference.
- **DB-connection lifecycle first exercise:** C6 (compute-layout) is the first Lambda that opens a psycopg2 connection. Per-invocation pattern; no pool. README notes this; C6's child session adds `psycopg2-binary` to compute-layout's `pyproject.toml` and writes the connection-management code.
- **Smoketest cleanup at C4:** C4's first commit deletes `python/lambdas/smoketest/` + drops the ECR repo `solarlayout/smoketest` (per v1.4 amendment to C4's Acceptance). The build-lambdas matrix workflow auto-discovers `python/lambdas/parse-kmz/` instead.

- Real Lambda business logic (C4 ships parse-kmz; C6 ships compute-layout; etc.).
- SQS local-emulation server (LocalStack, ElasticMQ) — not needed; SQS-shaped Lambdas use sync HTTP locally with the 202+thread async pattern.
- RDS Proxy / connection pooling — deferred to C6 if heavy load shows pgbouncer is needed.
- Lambda-invoker as a shared package (`packages/lambda-invoker/`) — stays in `apps/mvp_api/src/lib/` for v1; promote when a second consumer materializes (mobile, future web layout app).

- AWS Lambda RIE (explored + rejected per §2 + `feedback_proven_over_novel.md`).
- Dockerfile.local + docker-compose Lambda services (rejected per v1.6 amendment; bip-pipeline pattern is the chosen path).
