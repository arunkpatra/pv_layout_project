## 3. Design refinements adopted during the demo

### 3a. ICR drag must translate the polygon ring, not just a marker

**Observation:** first demo version rendered only a small vertex dot at the cursor during drag. User feedback: "dot moves but ICR stays put" — doesn't match PVlayout_Advance, which (per Agent 2's report) moves the ICR rectangle itself.

**Fix:** on mousedown, capture the hit ICR's polygon ring via `feat.geometry.coordinates[0]`. On mousemove, translate the ring by `(cursor - clickPoint)` and render via `kmz-draw-preview`. Commit payload reports `newCenter = originalCenter + dragDelta` rather than the mouseup pixel point — the latter is where the mouse ended up, not where the ICR centroid ended up.

**Spec impact:** already folded into spec §6 (icrDrag section) and spec §3 data-flow; no further amendment needed.

### 3b. Imperative preview > React-mediated preview

**Observation:** routing every mousemove through Zustand → React subscriber → useMemo → prop → MapCanvas effect → `hydrateSources` took 10-30ms per event. Each mouse event paid a full React render cycle; the preview lagged visibly behind the cursor and "danced around."

**Fix:** mode modules now call `setDrawPreview(map, previewFc, verticesFc)` directly — one function call that lands on `map.getSource('kmz-draw-preview').setData(...)`, no React involvement. Zustand still tracks mode / session start+end / undoStack (low-frequency semantic state); per-pixel preview geometry is a render-loop concern, owned by the mode modules.

**Spec impact:** spec §3 data-flow paragraph about "every setInProgressGeometry triggers re-render → setData" is now incorrect for the drag/draw hot path. Spec amended in this commit chain. Key rule added: **high-frequency canvas interactions (drag, rubber-band) bypass React and write directly to MapLibre sources; React owns semantic state only.**

This is a design lesson worth carrying into future spikes (S11 proper, plus any future high-frequency interaction work): **don't bounce 60Hz event streams through React's render cycle.**

## 4. LOC honesty

Actual ~2.1× the 440 LOC estimate. Breakdown:

```
apps/desktop/src/canvas/debug.ts                77
apps/desktop/src/canvas/coords.ts               66
apps/desktop/src/canvas/preview.ts              66  (added during imperative refactor)
apps/desktop/src/canvas/InteractionController   145
apps/desktop/src/canvas/modes/icrDrag.ts        199  (grew from 175 during ring-translate refactor)
apps/desktop/src/canvas/modes/rectDraw.ts       141  (grew from 131 during imperative refactor)
apps/desktop/src/state/editingState.ts          147  (grew from 142 with originalRing + originalCenter fields)
apps/desktop/src/panels/DrawingToolbar.tsx       81
─────────────────────────────────────────────────
Total new code                                  922

Diff to existing files (MapCanvas +27, App.tsx +94, style JSONs +41×2)   203
                                                ────
Grand total new / changed TS + JSON             ~1,125

Unit tests (coords + editingState)              211
```

**Why the overrun:** each file carries ~20-30 lines of module-level JSDoc (intentional — they document *why* each module exists + the non-obvious design rules like "bypass React on mousemove"). Defensive error paths in the mode modules (`try/catch` around `onCommit`, explicit cleanup in detach) also add ~30 LOC per mode. Neither is waste; both serve S11's "reviewable, fully testable, fully ours" promise.

**What this means for S11:** the ~440 estimate was the floor. The real ceiling is ~1,000 LOC if S11 picks up polygon + line + selection modes (each ~100-150 LOC) and adds the proper mutation wiring. We should be honest about that in the S11 SPIKE_PLAN entry if the original estimate leaks expectations.

An aggressive squash run (tighten JSDoc, consolidate cleanup) would shave maybe 150-200 LOC. Not worth it for a ~1,000 LOC module. The docs pay for themselves on the first bug someone else has to diagnose.

## 5. What S11 inherits

The demo branch is deleted; S11 re-implements from scratch. But S11 is not starting from zero — it inherits:

- **Proven design patterns:**
  - Mode modules as pure functions with an `attach(ctx) → detach` interface.
  - InteractionController owning the mode-switch lifecycle + styledata re-attach.
  - Zustand slice for mode / session / undoStack; direct-to-map `setData` for preview.
  - Style-JSON declarations for draw-preview + draw-vertex sources so theme swap is free.
  - Probe factory (`canvas/debug.ts`) for structured dev-time observability.
  - Coords helpers (`canvas/coords.ts`) with Haversine distance and rect-ring builder.

- **A concrete UX pattern for commits** (added during the demo debrief; see §6 below and ADR-0006 Consequences): preview geometry persists on the canvas between drag-release and sidecar ack, showing "pending" state during the round-trip. Mode transitions from `drag-icr` / `draw-*` to a new `awaiting-ack` state, not back to `idle`, until the mutation settles.

- **Parity semantics already pinned** (spec §9 P1–P11): 1m² rect minimum (sidecar-enforced), 10m polygon close tolerance in Haversine metres (not component-wise UTM), drag bounds against `usable_polygon`, LA-before-inverter refresh order on any commit, drag/draw mutual exclusion, cable staleness after ICR drag acknowledged as intentional.

- **A working reference implementation** — the demo branch code can be read (until deleted) for pattern reference. The lessons survive in the spec + ADR + this memo; the code is disposable.
