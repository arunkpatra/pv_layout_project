## What was built

A standalone, self-contained `pvlayout-engine` binary. All domain and HTTP dependencies (matplotlib, shapely, pyproj, ezdxf, FastAPI, uvicorn, etc.) are bundled inside a single ~52 MB file. The binary runs the same FastAPI sidecar as `uv run python -m pvlayout_engine.main` does, with the same `READY {...}` stdout contract, the same `/health`, `/parse-kmz`, `/layout`, `/refresh-inverters` routes, and the same bearer-token auth.

### 1. The PyInstaller spec — `python/pvlayout_engine/pvlayout-engine.spec`

Single-file, console-mode spec. Key choices:

- **`collect_all()` for matplotlib, pyproj, shapely, ezdxf, simplekml.** These libraries ship non-Python data files (font atlases, projection databases, DXF templates) and have dynamic imports that PyInstaller's static analysis doesn't catch by itself.
- **Explicit uvicorn hiddenimports.** Uvicorn picks its HTTP + WebSocket + event-loop implementations at runtime via `auto` modules; every candidate has to be listed explicitly or the bundle silently drops them.
- **First-party `collect_submodules` for `pvlayout_core` and `pvlayout_engine`.** Belt-and-braces — walks every submodule even if an import path is missed.
- **Explicit excludes** for `PyQt5`, `PyQt6`, `PySide2`, `PySide6`, `tkinter`, `PIL.ImageTk`, `IPython`, `notebook`, `jupyter`. Stops transitive pulls from bloating the binary and enforces the "no GUI libs" contract.
- **`console=True`** on every platform. Tauri (S5) will spawn this binary via its Command API with stdio pipes; console subsystem on Windows guarantees stdout is connected even when CreateNoWindow is set.

### 2. Smoke script — `scripts/smoke_binary.sh`

A plain POSIX-compliant shell script that:
1. Launches the binary with a fixed dev token.
2. Polls stdout for up to 30 s waiting for the `READY {...}` line.
3. Parses the port out of it.
4. Curls `/health` with the token — asserts 200.
5. Curls `/health` without the token — asserts 401.
6. Cleans up on exit (`trap` kills the child, removes temp files).

Used locally right now; invoked by the CI workflow later.

## Known limitations (by design or flagged)

- **Cold-start time: ~10 s on macOS, ~5 s on Linux, ~3–5 s on Windows.** `--onefile` unpacks the ~50 MB archive into a per-invocation temp dir before running. This is PyInstaller's standard onefile tax. It's acceptable for S4's standalone-binary gate but is a **real UX concern for S5** (Tauri will block on sidecar readiness before the first React call can succeed).

  Three options for S5+, ordered by preference:
  1. **Stay on `--onefile`, paper over with a splash screen.** Tauri shows a branded splash while the sidecar unpacks. ~1 week of net UX loss is hidden.
  2. **Switch to `--onedir`.** Binary startup becomes ~1 s, but we ship a folder instead of a file. Tauri bundling handles either. This is a single `spec` change.
  3. **Precompute `_MEIPASS` at install time.** Use `--runtime-tmpdir`. Advanced, brittle.

  I'll default to option 1 (splash screen) and keep onefile unless you want option 2 pre-emptively. Flag this in S5 kickoff.

- **`version` in the bundled binary is literally `"0.0.0"`.** The `_git_short_sha()` helper looks for a `.git` directory walking up from the running script; in a PyInstaller bundle, the script lives inside `_MEIPASS`, which never has one. This is **correct** — S14's release pipeline will inject the real version via the `PVLAYOUT_VERSION` env var at bundle time.

- **Binary is host-specific.** The `dist/pvlayout-engine` I just built is arm64 Mach-O and runs only on Apple Silicon Macs. Cross-compilation is a CI-matrix concern (each runner builds its own); there's no sane way to cross-build PyInstaller bundles locally.

- **CI workflow is scaffolding, not proof.** The first push to a GitHub remote will validate (or reveal issues in) the Windows/Linux/macOS-x64 builds. PyInstaller hiddenimports are platform-sensitive — expect small iteration on the spec the first time CI runs. Not a blocker for S4 local gate; just transparent planning.

- **No code signing, no notarization.** That's S14. Unsigned binaries on macOS Sequoia/Sonoma will trigger Gatekeeper on first run; Windows will hit SmartScreen. Fine for dev.

## Decisions made during this spike

1. **`--onefile` over `--onedir` for now.** Matches spike-plan prescription. The startup-time cost is real but well-understood and workaround-able; onedir as a fallback remains a one-line change.

2. **`console=True` on all platforms.** Tauri will consume our stdout via its Command API; console subsystem on Windows keeps that reliable. The cosmetic "console window flash" on Windows is a non-issue because Tauri spawns with `CREATE_NO_WINDOW`.

3. **`collect_all()` over manually hunting hiddenimports.** Slightly larger bundle, much lower risk of "missing submodule" failures on edge-case code paths (which is exactly what bit S1/S3 with the `utils.geo_utils` regression). Worth the 3–5 MB.

4. **CI paths-filter on `python/pvlayout_engine/**`.** UI-only and docs-only changes don't waste matrix runner time. Tight, matches `mvp_web`'s philosophy.
