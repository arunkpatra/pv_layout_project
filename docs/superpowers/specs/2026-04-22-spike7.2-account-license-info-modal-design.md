# Spike 7.2 — Account & License Info Modal: Design Spec

**Date:** 2026-04-22  
**Spike:** 7.2  
**Status:** Approved for implementation

---

## Goal

Give licensed users a way to view their account details (name, email, plan, features, remaining
calculations) and change their license key from within the desktop app.

---

## Architecture

Two parts:

1. **API** (`apps/mvp_api`) — extend `GET /entitlements` to return user profile and plan details
2. **Python app** (`PVlayout_Advance`) — new dedicated app toolbar + `LicenseInfoDialog`

No new endpoints. No DB changes. `ProductFeature.label` already stores human-readable feature
labels.

---

## Part 1 — API: Extended `GET /entitlements` Response

### Query changes

The route already resolves the User via `licenseKeyAuth` middleware. It needs two additions:

- Include `user.name` and `user.email` from the `User` record (populated at first Clerk auth)
- Include `Entitlement` records joined to `Product` + `ProductFeature`, ordered by
  `product.displayOrder`

### Response shape

Top-level fields (`licensed`, `availableFeatures`, `totalCalculations`, `usedCalculations`,
`remainingCalculations`) are **unchanged** — the Python status bar still works without
modification.

New fields added under `data`:

```json
{
  "success": true,
  "data": {
    "user": {
      "name": "Ravi Kumar",
      "email": "ravi@example.com"
    },
    "plans": [
      {
        "planName": "PV Layout Pro",
        "features": [
          "Plant Layout (MMS, Inverter, LA)",
          "Obstruction Exclusion",
          "AC & DC Cable Routing",
          "Cable Quantity Measurements"
        ],
        "totalCalculations": 10,
        "usedCalculations": 3,
        "remainingCalculations": 7
      }
    ],
    "licensed": true,
    "availableFeatures": ["plant_layout", "cable_routing"],
    "totalCalculations": 10,
    "usedCalculations": 3,
    "remainingCalculations": 7
  }
}
```

`plans[]` is ordered by `product.displayOrder`. A user with multiple purchases will have
multiple entries. Features within each plan are ordered by `ProductFeature` insertion order
(seed order).

### Null handling

`user.name` may be null if Clerk did not provide a name at signup. The API returns `null`;
the Python dialog renders "—" in that case.

### Tests

- Existing test: `GET /entitlements` returns 200 with basic fields — update to assert new fields
  present
- New test: `user` block contains `name` and `email` from the User record
- New test: `plans` block contains correct plan name, labels (not slugs), and calculation counts
- Existing 401 test unchanged

## Part 2 — Python App

### New dedicated app toolbar (`main_window.py`)

A `QToolBar` added via `QMainWindow.addToolBar(Qt.TopToolBarArea, ...)`, positioned above the
matplotlib `NavigationToolbar`. This is the app-level toolbar — separate from matplotlib canvas
controls. `setMovable(False)`.

Layout: expanding spacer on the left pushes the user button to the right end.

```
[──────────────────────────── spacer ────────────────────────────] [👤]
```

The user button (`QToolButton`) uses a bundled person/user icon from Qt's standard icon set or
a small PNG asset. It is:
- **Visible** when `self._license_key` is not None
- **Hidden** when no key is stored (first-run users haven't linked an account)

Visibility is toggled by `_show_user_button()` and `_hide_user_button()` helpers, called from
the existing banner / entitlement result handlers.

### `_on_user_btn_clicked()`

1. Disable the user button (prevents double-click)
2. Set `_pending_info: bool = True`
3. Fire `_start_entitlements_check()` — re-uses existing `EntitlementsWorker`
4. `_on_entitlements_result` detects `_pending_info`, opens `LicenseInfoDialog(self, data)`,
   clears `_pending_info`, re-enables the user button
5. `_on_entitlements_error`: clears `_pending_info`, re-enables button, shows brief status bar
   message ("Could not load account info — check your connection")

Always re-fetches from the API on every open. The screen may have been idle for hours.

### New state variable

```python
self._pending_info: bool = False  # True while re-fetching for info dialog
```

### `gui/license_info_dialog.py` — `LicenseInfoDialog`

New file. Read-only modal populated from the fresh `_entitlements` dict passed in at construction.
No network calls inside the dialog.

**Layout (top to bottom):**

```
Account
───────────────────────────────
Name:   Ravi Kumar
Email:  ravi@example.com

Your Plans
───────────────────────────────
PV Layout Pro
  Features: Plant Layout (MMS, Inverter, LA), Obstruction Exclusion,
            AC & DC Cable Routing, Cable Quantity Measurements
  7 of 10 calculations remaining

PV Layout Basic          ← shown only if user has multiple purchases
  Features: ...
  3 of 5 calculations remaining

───────────────────────────────
              [Change Key]  [Close]
```

- Plan sections are rendered in a loop — one `QGroupBox` per plan entry
- Features are displayed as a comma-separated string (wraps if long)
- "X of Y calculations remaining" — X in red bold if 0, gray otherwise
- If `user.name` is null, render "—"

**Change Key flow:**

```python
def _on_change_key(self):
    dlg = LicenseKeyDialog(self)
    if dlg.exec_() == QDialog.Accepted:
        self._main_window._license_key = dlg.get_key()  # tentative
        self._main_window._start_entitlements_check()
        self.close()
```

New key is only persisted to the OS credential store after `_on_entitlements_result` confirms it
valid (existing validate-then-persist rule, unchanged from Spike 7).

If the new key is invalid (401), the banner reappears in `main_window`, the dialog is already
closed, and the old key is cleared — same flow as first-run invalid key entry.

### `_on_entitlements_result` changes

Existing handler gains one branch:

```python
if self._pending_info:
    self._pending_info = False
    self._user_btn.setEnabled(True)
    dlg = LicenseInfoDialog(self, data)
    dlg.exec_()
```

This branch runs **in addition to** the existing logic (saving key, updating quota label,
handling `_pending_generate`). Order: existing logic first, then `_pending_info` branch.

### Icon

Use `QStyle.SP_DialogApplyButton` or a bundled 20×20 PNG. A bundled PNG is preferred for
consistency across platforms — place at `assets/user_icon.png` and load via
`QIcon("assets/user_icon.png")`.

## Error Handling

| Scenario | Behaviour |
|---|---|
| Network error opening modal | Button re-enabled; status bar: "Could not load account info — check your connection" |
| 401 on modal open | Key cleared; banner shown; button hidden (no key stored) |
| `user.name` is null | Show "—" in Name field |
| User has zero plans | `plans[]` is empty; show "No active plans." in plans section |
